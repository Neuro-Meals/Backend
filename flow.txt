NutrioMeals Business Process

Below is the complete business flow your backend currently supports, from user registration through subscription, payment, orders, delivery, administration, nutrition management, and reporting.

1. Customer Registration and Authentication
Step 1: Customer creates an account

The frontend sends:

POST /auth/register

The user provides information such as:

{
  "first_name": "Adam",
  "last_name": "Katani",
  "email": "adam@example.com",
  "phone": "+255700000000",
  "password": "StrongPassword123",
  "location": "Dar es Salaam",
  "address": "Kinondoni",
  "gender": "male",
  "age": 25,
  "height_cm": 175,
  "weight_kg": 70,
  "fitness_goal": "weight_loss",
  "dietary_preference": "high_protein",
  "allergies": ["nuts", "dairy"]
}

The backend:

Checks whether the email already exists.
Checks whether the phone number already exists.
Hashes the password.
Creates the user with the customer role.
Generates a secure OTP.
Stores the OTP and expiration time.
Sends the OTP to the customer’s email.
Marks the account as unverified.

The user cannot fully access the system until the email is verified.

Step 2: Customer verifies email

The frontend sends:

POST /auth/verify-email

Payload:

{
  "email": "adam@example.com",
  "otp": "123456"
}

The backend:

Finds the user.
Checks whether the OTP matches.
Checks whether the OTP has expired.
Marks the user as verified.
Removes the used OTP.

After this, the user can log in.

Step 3: Resend verification OTP

When the OTP expires or is not received:

POST /auth/resend-verification-otp

The backend:

Finds the account.
Generates a new OTP.
Sets a new expiration time.
Sends a new verification email.

This endpoint should be protected by rate limiting.

Step 4: Customer logs in

The frontend sends:

POST /auth/login

Payload:

{
  "email": "adam@example.com",
  "password": "StrongPassword123"
}

The backend:

Finds the user.
Verifies the password.
Confirms that the account is verified.
Generates a JWT access token.
Returns the user profile.
Returns the user role.
Returns assigned permissions.
Returns the latest active or pending subscription.

The response can include:

{
  "access_token": "...",
  "user": {
    "id": 12,
    "first_name": "Adam",
    "last_name": "Katani",
    "email": "adam@example.com",
    "role": "customer",
    "permissions": [],
    "subscription": null
  }
}

The frontend stores the JWT and sends it with protected requests:

Authorization: Bearer ACCESS_TOKEN
Step 5: Customer views own account
GET /auth/me

This returns:

Personal details
Role
Permissions
Fitness details
Allergies
Current subscription
Step 6: Customer updates profile
GET /profile/
PUT /profile/

The user can update information such as:

Location
Delivery address
Weight
Height
Fitness goal
Dietary preference
Allergies

This information can later help nutrition managers choose suitable meals.

2. Password Recovery
Forgot password
POST /auth/forgot-password

The backend:

Accepts an email.
Returns a generic response whether the account exists or not.
Generates a password-reset OTP for an existing account.
Sends the OTP by email.
Stores an expiration time.

Recommended response:

{
  "message": "If the account exists, a password reset email has been sent."
}
Reset password
POST /auth/reset-password

Payload:

{
  "email": "adam@example.com",
  "otp": "123456",
  "new_password": "NewStrongPassword123"
}

The backend:

Checks the OTP.
Checks expiration.
Hashes the new password.
Replaces the old password.
Clears the reset OTP.
Change password while logged in
POST /auth/change-password

The user provides:

Old password
New password

The backend verifies the old password before updating it.

10. Stripe Payment Process
Step 1: Customer creates checkout
POST /payments/create-checkout

Payload:

{
  "subscription_id": 5
}

The backend:

Verifies that the subscription belongs to the logged-in user.
Verifies that it is not already paid.
Creates a local payment record.
Sets payment status to pending.
Creates a Stripe Checkout Session.
Returns a Stripe checkout URL.

Response:

{
  "payment_id": 8,
  "checkout_url": "https://checkout.stripe.com/...",
  "stripe_checkout_session_id": "cs_test_..."
}
Step 2: Frontend redirects to Stripe

The customer opens checkout_url.

Stripe collects card details securely.

The backend does not store card numbers.

Step 3: Stripe payment succeeds

Stripe redirects the user to:

FRONTEND_SUCCESS_URL

Example:

https://nutriomeals.com/payment-success
?session_id=cs_test_xxx
&payment_id=8
Step 4: Frontend verifies payment

The frontend reads session_id and calls:

GET /payments/verify-session/{session_id}

The backend:

Retrieves the Checkout Session from Stripe.
Confirms payment_status == paid.
Finds the local payment.
Sets payment status to paid.
Saves the Stripe payment intent ID.
Sets paid_at.
Updates the subscription:
status = active
payment_status = paid
Sets subscription start and end dates.
Step 5: Stripe webhook backup
POST /payments/webhook/stripe

The webhook performs the same payment activation even if the customer closes the browser before returning to the frontend.

This makes payment processing reliable.

Step 6: Customer views own payments
GET /payments/my

The customer sees their payment history.

Step 7: Admin views all payments
GET /payments/

The response includes:

Customer details
Subscription details
Plan name
Payment provider
Amount
Currency
Payment status
Paid date
Stripe references

Finance managers can filter paid, failed, or pending payments.

12. Order Creation Process

An order should only be created after successful payment.

Endpoint:

POST /orders/from-subscription

The backend verifies:

subscription.status = active
subscription.payment_status = paid

It then:

Loads the subscription plan.
Checks that no active order already exists.
Generates an order number.
Creates an order.
Sets status to pending.
Sets the delivery date.
Copies the delivery address.
Stores plan details inside the order items.

Example order number:

ORD-20260710-A1B2C3D4

17. Driver App Flow
Step 1: Driver logs in

The driver uses the normal login endpoint:

POST /auth/login

The returned role is:

driver

The frontend opens the driver dashboard.

Step 2: Driver views assigned deliveries
GET /driver/deliveries

The backend filters:

Delivery.driver_id == current_user.id

Therefore, each driver only sees their own deliveries.

Step 3: Driver views one delivery
GET /driver/deliveries/{delivery_id}

The backend confirms that the delivery belongs to that driver.

Step 4: Driver picks up the order
POST /driver/deliveries/{delivery_id}/pickup

The backend:

Changes delivery status to picked_up.
Stores pickup time.
Updates order status.
Step 5: Driver starts delivery
POST /driver/deliveries/{delivery_id}/out-for-delivery

The backend changes:

delivery.status = out_for_delivery
order.status = out_for_delivery
Step 6: Driver sends location
PATCH /driver/deliveries/{delivery_id}/location

Payload:

{
  "latitude": -6.7924,
  "longitude": 39.2083
}

The customer frontend can use these coordinates for live tracking.

Step 7: Driver completes delivery
POST /driver/deliveries/{delivery_id}/complete

The backend:

Sets delivery status to delivered.
Stores delivery time.
Sets order status to delivered.
Step 8: Driver reports failure
POST /driver/deliveries/{delivery_id}/fail?reason=Customer unavailable

The backend:

Sets delivery status to failed.
Stores the failure reason.
